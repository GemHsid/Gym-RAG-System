# 24 小时无人值守健身房 AI 智能客服系统

本项目面向 24 小时夜间无人值守健身房场景，构建基于 Advanced RAG 的 AI 智能客服系统。系统以微信小程序为用户入口，以 Django 微服务为统一业务中台，以 GPU 推理服务承载检索增强问答能力，目标是在夜间低人力、无人值守、用户问题高频发生的运营环境中，为场馆提供可持续演进的智能客服解决方案。

本文档已结合 [论文.md](file:///d:/毕设整合/论文.md) 的相关内容补充业务背景、技术逻辑与验证口径，并对项目定位、技术架构、风险兜底与指标体系进行统一整理。

## 1. 项目概述

### 1.1 系统定位

系统定位不是通用问答机器人，而是服务于 24 小时无人值守健身房的垂直领域 AI 智能客服系统。其核心目标是在夜间或非人工值守时段，承接用户高频咨询、标准业务查询与基础流程办理，降低场馆人工客服压力，同时保障回答准确性、服务连续性与风险可控性。

### 1.2 业务目标

- 降低夜间无人值守场景下的人工值守成本
- 承接场馆高频重复咨询，提升用户问题响应效率
- 为会员提供可随时访问的标准化服务入口
- 将客服能力从单点问答扩展为可持续优化的知识服务体系

### 1.3 产品目标

- 以“回答准确性优先、风险可控”为首要原则
- 以高频、标准化、低风险问题为第一阶段落地重点
- 通过可追溯的检索与会话机制支撑持续迭代
- 打通 AI 客服、业务系统、后台管理、人工接管之间的完整闭环

### 1.4 场景分层方法

系统围绕以下四个维度对客服场景进行分层：

- 问题频次
- 标准化程度
- 风险等级
- 闭环价值

基于以上维度，优先落地以下高频、标准化、低风险场景：

- 场馆基本信息导览
- 会员套餐咨询
- 会员权益查询
- 退款业务办理
- 器械使用规范解答

在演进路径上，系统由“单点问答能力”逐步发展为“可持续演进的 RAG 服务体系”，实现从 FAQ 检索、知识库问答到多场景客服编排与持续运营优化的升级。

## 2. 技术架构与工作流设计

### 2.1 核心设计原则

系统在产品与技术实现上始终坚持以下红线：

- 回答准确性优先
- 风险可控优先
- 可追溯优先于黑盒生成
- 在无人值守场景下优先保证稳定性与可兜底性

### 2.2 技术选型

项目采用以下核心技术组合：

- 微信小程序：提供用户交互入口
- Django + DRF：提供统一微服务接口与业务编排
- LlamaIndex：承载 Advanced RAG 核心编排
- Qwen3-Embedding-8B：负责向量化表示
- Qwen3-8B：负责最终回答生成
- PostgreSQL：存储结构化业务数据
- Milvus：存储向量索引
- Django Admin：承载后台运营与人工管理
- 沐曦 曦云 C500 GPU：承载模型推理与服务部署

### 2.3 Advanced RAG 全流程链路

系统围绕无人值守健身房知识问答场景，完整实现了以下技术链路：

1. Unicode 文本清洗  
   对原始知识文档进行统一字符规范化、脏数据处理与文本净化，降低切块与检索噪声。

2. 层级切块  
   采用 `[1536, 512, 256]` 的层级切块策略，对知识内容进行粗粒度到细粒度的递进式分片，兼顾全局语义覆盖与局部检索精度。

3. 双路召回  
   同时执行向量检索与 BM25 稀疏检索，确保既能覆盖语义相似问题，也能保留关键词匹配能力。

4. RRF 融合  
   使用 RRF（Reciprocal Rank Fusion）融合双路召回结果，降低单一检索策略的偏差。

5. 召回索引回溯  
   对召回结果进行索引级回溯与来源检查，增强过程可解释性与来源可追踪性。

6. 精排与上下文组织  
   使用 `LLMRerank Top7` 对候选片段进行重排，最终拼接出更高质量的上下文输入大模型。

7. 最终生成  
   由 `Qwen3-8B` 基于检索上下文生成回答，输出给小程序端并同步进入后端会话记录体系。

### 2.4 工作流优化重点

系统从以下四个维度持续优化上下文处理逻辑与召回策略：

- 意图理解准确性
- 知识完整性
- 输出一致性
- 过程可追溯性

该设计使系统不仅能“回答问题”，还能够解释答案来源、回溯召回过程，并为后续知识库优化提供依据。

## 3. 风险兜底与人工协同机制

无人值守场景对客服系统最大的要求不是“尽量回答”，而是“在不确定时安全地不乱答”。因此系统专门设计了风险兜底与人工协同机制，确保 AI 不会在高风险场景下越权输出。

### 3.1 兜底触发场景

系统针对以下四类场景设置标准化拦截规则：

- 低置信度输出
- 工具调用失败
- 多轮对话未解决
- 高风险对话

### 3.2 标准化处理策略

在触发拦截后，系统不直接继续自由生成，而是执行以下策略：

- 输出保守型回复
- 提示用户转人工或走标准业务入口
- 标记该轮对话为需人工关注
- 对异常链路进行后台追踪与后续复盘

### 3.3 人工协同要求

在执行人工转接前，系统要求自动结构化沉淀以下信息：

- 用户核心意图
- 完整历史对话记录
- 已执行的全部操作步骤
- 当前失败原因或拦截原因

该机制可以保证服务链路无断点，使人工客服无需重复追问基础信息，即可直接接手处理，提高协同效率并降低用户流失风险。

## 4. 指标体系与效果验证标准

### 4.1 四维指标体系

为确保系统不仅可用，而且可量化评估，项目建立了覆盖四个层面的指标体系：

- 业务层：问题解决率、人工转接率、夜间服务覆盖率、业务闭环完成率
- 能力层：召回率、答案忠实度、回答相关度、上下文命中质量
- 风险层：高风险误答率、低置信度拦截准确率、异常兜底成功率
- 系统层：接口可用性、平均响应时延、服务稳定性、日志可追溯完整率

### 4.2 离线效果验证

项目采用 RAGAS 框架对 RAG 问答效果进行离线验证，并以“召回能力 + 忠实度 + 回答相关度”作为核心评估主指标。

最终验证结果为：

- 召回率提升 22%
- 回答忠实度提升 30%
- 回答相关度提升 6%

该结果说明系统在知识召回、答案贴合原始知识、以及对用户问题的响应相关性上均获得了可量化提升，为无人值守客服场景的稳定落地提供了依据。

## 5. 项目整体业务背景

虽然本文档聚焦 AI 智能客服主线，但项目整体并非孤立的问答 Demo，而是建立在完整健身房数字化系统之上的智能化服务模块。项目还包含以下背景业务能力：

- 微信小程序用户端
- 会员体系与个人中心
- 课程预约与防超卖并发控制
- 会员卡、订单与退款流程
- 器械展示与使用说明
- Django Admin 后台管理

这些业务模块为 AI 客服提供了真实的业务上下文与知识来源，使其能够围绕场馆信息、会员权益、课程咨询、退款规则、器械规范等内容进行可落地回答，而不是停留在通用闲聊层面。
